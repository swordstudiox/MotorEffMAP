1. 项目定位

MotorEffMAP 是一个用于绘制电驱系统效率 MAP 的 Python 桌面程序。用户通过 Excel 导入电机/电控测试数据，通过 MotorEffMAP.ini 配置列名和绘图参数，程序批量输出 MCU、电机、系统效率 MAP，以及效率区域占比结果。

  • GUI：PySide6。
  • 数据读取：pandas、openpyxl、xlrd。
  • 数值计算：numpy、scipy。
  • 绘图：matplotlib。

7. 输入 Excel 要求

程序使用 pandas.read_excel(file_path, sheet_name=None) 读取所有 sheet。每个 sheet 的第一行会作为列名。

  • 第一行是列名，并且列名与 MotorEffMAP.ini 中配置完全一致。
  • 转速、扭矩、功率、效率、电压列应能转换为数字。
  • 可以存在空行，程序会在归一化阶段删除核心列为空的数据。
  • 可以存在单位行或文本行，只要核心列无法转成数字，就会变成 NaN 并被删除。
  • 同一 sheet 建议只包含同一转向、同一电动/发电状态、同一电压等级的数据。

基础必需列是转速、扭矩、功率和电压；如果填写了有效 customUdc，电压可以不来自 Excel 列。效率列是可选联动项，Eff_MCU、Eff_Motor、Eff_SYS 至少有一个可用即可继续。

图 3：基础列缺失会停止处理；效率列按配置和开关联动。

图 3 读图路线：Excel 需要哪些列才算能跑？

这张图回答的是：哪些列是基础必需列，哪些效率列是可选列，以及为什么不能要求 MCU、电机、系统效率三列全部填写。

1先查基础列

Speed、Toqrue、P_Motor 和电压来源必须可用。

缺基础列时没有坐标或功率，后续 MAP 无法成立。

2再查效率列

Eff_MCU、Eff_Motor、Eff_SYS 按配置填写情况判断。

填了哪个、且对应开关开启，就处理哪个。

3最后决定输出

至少一个效率类型可用即可继续。

不可用类型只跳过，不影响其它类型输出。

重点：可选效率列不能进入全局必填链，否则一个空配置会阻断其它正常图。

8. 逻辑层实现

8.1 数据加载

self.sheets_dict = pd.read_excel(file_path, sheet_name=None)

返回结构是 {sheet_name: DataFrame}。GUI 会把每个 sheet 加入文件列表，处理时通过 set_current_sheet(sheet_name) 切换当前 DataFrame。读取阶段只拆工作簿，不做列名判断和数值清洗，错误会定位到具体 sheet。

8.2 列映射和基础判断

filter_data() 会去掉列名首尾空格，用 pd.to_numeric(..., errors='coerce') 转为数字，保留 NaN，不把空值填成 0。

图 4：列映射阶段只保留真实数据和 NaN，不把缺失值伪造成 0。

图 4 读图路线：INI 列名怎么变成程序内部字段？

这张图回答的是：配置里写的是 Excel 表头，程序内部使用统一字段名，中间必须做显式映射和数值化。

1读取配置列名

例如 Speed = 转速、Eff_Motor = 电机效率。

先去掉空格和引号，再到 Excel 表头中匹配。

2转换为数值

使用 pd.to_numeric(errors='coerce')。

无法转换的内容变成 NaN，而不是变成 0。

3判断方向和状态

可自动根据扭矩、功率判断正反转和驱动/发电。

customSpeedDirection/customMotionState 填写后覆盖自动判断。

重点：缺失值保留为 NaN 是为了让清洗阶段明确删除；伪造成 0 会污染包络线和面积占比。

判断项规则
转向转速均值 > 0 为 正转，否则为 反转。
状态功率均值 > 0 为 电动，否则为 发电。
效率列条件行为
配置项为空跳过该效率类型。
配置项有值且对应输出开启列必须存在，否则当前 sheet 失败。
配置项有值但输出关闭尝试读取；失败只警告，不阻断其它输出。
三个效率列都不可用当前 sheet 失败，因为没有可计算对象。

8.3 归一化

normalization() 删除基础列和当前可用效率列存在 NaN 的行，将相邻差值不超过 6 rpm 的转速合并为平均转速，再过滤效率值，只保留 [0, 100)。

图 5：归一化只处理当前实际可用的效率列，避免未填写列阻断其它图。

图 5 读图路线：清洗数据时到底删什么、保留什么？

这张图回答的是：为什么程序先删核心列空行，再做转速合并和效率范围过滤，而不是直接拿原始表插值。

1删除核心空行

只按坐标、功率、电压和当前可用效率列删除 NaN。

未配置的效率列不参与清洗。

2合并转速层

相邻转速差小于阈值时归入同一层。

这样包络线不会因为台架波动产生毛刺。

3过滤效率范围

效率必须满足 0 <= eff < 100。

过滤后按 Speed/Torque 排序，给后续分组和插值使用。

重点：归一化的目标是得到可信的 processed_df，它是包络线、插值和占比的唯一数据源。

8.4 外特性包络线

程序按转速分组取最大扭矩，再根据点数选择常数曲线、线性插值或 PchipInterpolator 保形插值。插值结果会限制在 [0, 观测最大扭矩 * 1.05]，防止过冲导致网格异常变大。

图 6：包络线决定效率 MAP 的几何运行区域，不让网格扩展到无意义的高扭矩区域。

图 6 读图路线：外特性包络线从哪里来？

这张图回答的是：为什么要先算包络线，以及包络线如何限制后续网格、显示和面积占比的边界。

1按转速分组

清洗后的数据先按 Speed 分组。

每一组代表一个转速层。

2取最大扭矩

每个转速层取 Torque.max()。

这些点就是外特性采样点。

3插值成连续边界

1 点用常数、2 点用线性、多点用 PCHIP。

结果裁剪到 0 ~ 最大观测扭矩 * 1.05。

重点：包络线不是效率等高线，它是“可运行区域上边界”；没有它，网格会扩展到物理上没有数据支撑的区域。

8.5 网格和插值

转速轴从 0 到最大转速按 SpeedGrid 生成。扭矩轴按每个转速列独立生成，从 0 到外特性包络线扭矩，再通过 griddata(..., method='linear') 插值效率和功率。

XI, YI, ZI_Power, ZI_Eff, mask_valid_geo = process_map_data("Eff_MCU")
图 7：程序生成的是包络线内部的非矩形网格，矩阵空位用 NaN 表示。

图 7 读图路线：为什么网格不是普通矩形？

这张图回答的是：MAP 图虽然用二维矩阵绘制，但真实统计区域是按每个转速列独立生成的非矩形区域。

1生成转速轴

从 0 到最大转速按 SpeedGrid 建列。

每一列对应一个固定转速。

2逐列生成扭矩点

用 f_edge_curve(speed) 得到该列最大扭矩。

从 0 到最大扭矩按 TorqueGrid 建点，并补齐边界点。

3NaN 填充短列

不同列高度不同，短列空位填 NaN。

NaN 是矩阵占位，不属于几何区域。

重点：mask_valid_geo 只标记包络线内且未截止的网格点，它才是面积占比分母。

变量含义用途
points原始有效点的 (Speed, Torque) 坐标。插值输入坐标。
XI每个网格点的转速坐标。X 轴绘图、起始转速截止。
YI每个网格点的扭矩坐标，包络线外为 NaN。Y 轴绘图、几何区域判断。
ZI_Eff插值后的效率矩阵。效率填色、等高线、占比分子。
ZI_Power插值后的功率矩阵。功率等高线。
mask_valid_geo包络线内且未被起始坐标屏蔽的几何区域。效率占比分母。
点数保护唯一转速/扭矩点少于 3 个时，不调用 griddata，直接提示有效点不足。
维度保护点集共线或退化时直接失败，避免底层 Qhull 错误外泄。
网格保护max_rows * n_cols 超过 MaxGridPoints 时停止，避免内存异常。

8.6 效率区域占比

当前 GUI 使用 mask_valid_geo 作为分母，因此效率区域占比按几何运行区域计算。默认 StartSpeed=0、StartTorque=0 时，从 0rpm / 0Nm 开始。

Ratio(level) = count((ZI_Eff >= level) & mask_valid_geo) / count(mask_valid_geo) * 100
图 8：分母按几何运行区域，分子按几何区域内达到阈值的效率点。

图 8 读图路线：面积占比的公式怎么落到矩阵上？

这张图回答的是：同一个效率阈值下，哪些点进入分母，哪些点进入分子，NaN 和凸包外区域如何处理。

1先定分母

denominator = count(mask_valid_geo)。

它表示包络线内、起始坐标以上的几何网格点。

2再定分子

count((ZI_Eff >= level) & mask_valid_geo)。

只有几何区域内达标的效率点才算分子。

3最后算百分比

Ratio = 分子 / 分母 * 100。

每个 EffMAPStep 阈值都重复一次。

重点：分母不用“插值成功点数”，而用几何运行区域；这样凸包外无法插值的区域不会把占比虚高。

分母不是 griddata 成功插值的非 NaN 点数，而是包络线内、起始坐标以上的几何网格点数。凸包外但仍处于包络线内的点会计入分母；如果无法插值得到效率值，它不会计入分子，因此占比结果更保守。

8.7 核心算法原理图

下面三张图用几何区域解释第 8 章最容易误解的部分：程序统计的是包络线内的运行区域，不是完整矩形；效率达标区域是这个运行区域中的子集；StartSpeed 和 StartTorque 会同时改变图形显示起点和占比分母。

8.7.1 非矩形运行区域

图 9：效率 MAP 的统计区域不是完整矩形，而是外特性包络线以内的非矩形运行区域。

图 9 读图路线：分母区域为什么长成这个形状？

这张图回答的是：统计区域由电机外特性决定，不能简单用最大转速 × 最大扭矩的矩形代替。

1矩形区域只是画布

最大转速和最大扭矩只能定义坐标范围。

矩形右上角很多点并不是可运行点。

2包络线给出上边界

每个转速的最大扭矩不同。

包络线以下才是可以参与统计的几何区域。

3网格点代表面积

程序用网格点计数近似面积。

网格越细，占比越接近连续面积结果，但计算量越大。

重点：SpeedGrid/TorqueGrid 是面积近似精度和运行速度之间的平衡，不是随便越小越好。

8.7.2 面积占比分母和分子

图 10：分母是几何运行区域，分子是该区域内达到指定效率阈值的点，最终得到面积占比。

图 10 读图路线：高效区占比到底在比较哪两个面积？

这张图回答的是：以 90% 为例，“高效区”不是整张图上的蓝色区域，而是几何运行区域内部的达标子区域。

1底图：几何运行区域

包络线内的全部有效网格点构成分母。

起始坐标以下的点已经被剔除。

2叠加：达标区域

在底图内检查 ZI_Eff >= 90。

达标点构成分子，不达标点只留在分母中。

3曲线：多个阈值

80、85、90、95、99 等阈值分别计算。

阈值越高，达标区域通常越小，占比曲线下降。

记忆口径：先圈定“能运行的地盘”，再数“达到阈值的地盘”，最后相除。

8.7.3 起始坐标截止区域

图 11：设置起始转速或起始扭矩后，低于起点的区域会从显示和占比分母中同时剔除。

图 11 读图路线：StartSpeed/StartTorque 改了什么？

这张图回答的是：起始转速和起始扭矩不是只改坐标轴显示，它们也会改变面积占比的分母。

1生成截止掩码

cutoff_mask = (XI < StartSpeed) | (YI < StartTorque)。

低转速或低扭矩任一条件命中都截止。

2屏蔽图形矩阵

截止区域的 ZI_Eff 和 ZI_Power 设为 NaN。

因此 MAP 图不会显示这部分。

3同步调整分母

mask_valid_geo = 包络线内 & ~cutoff_mask。

占比只按保留区域统计。

重点：如果设置 StartSpeed=50、StartTorque=5，图形起点和占比分母都会从这个坐标开始。

9. 绘图实现

switch_plot() 负责绘制 MCU、电机或系统效率 MAP。效率使用 contourf 填色，效率等高线为黑色，功率等高线为绿色。保存 PNG 时使用约 25cm x 20cm 的图像比例和 200 DPI。

版式项当前实现
图像比例约 25cm x 20cm，对应 9.84 x 7.87 英寸。
GUI 显示容器使用与导出图一致的长宽比例。
边距通过 apply_figure_layout() 统一设置。
坐标起点从 StartSpeed / StartTorque 开始显示。
保存 DPI200。
图 12：MAP 图由效率填色、效率等高线和功率等高线叠加生成，显示和保存共用版式。

图 12 读图路线：一张 MAP 图由哪些图层叠出来？

这张图回答的是：用户看到的效率 MAP 不是单一图片，而是效率填色、效率等高线、功率等高线和标题信息的组合。

1底层：效率填色

contourf(XI, YI, ZI_Eff) 生成效率色块。

NaN 区域不着色，保留几何边界。

2中层：等高线

黑色线表示效率等级，绿色线表示功率等级。

标签会做重叠过滤，避免文字压在一起。

3输出：显示和保存

GUI 显示和 PNG 保存共用比例常量。

保存使用固定尺寸和 200 DPI，保证版式稳定。

重点：界面图和保存图必须共享长宽比与边距，否则会出现“保存正常、界面变形”的问题。

process_area_ratios() 会保存效率区域占比 Excel 和占比 PNG。占比图中 MCU 使用蓝色星标线，电机使用绿色圆点线，系统使用洋红色加号线。

占比图的保存和显示共用 _plot_ratio_on_axes()，数据来源共用 _collect_ratio_data()。因此手动点击“效率占比”和批处理保存 PNG 使用同一套开关判断、同一套比例、同一套曲线样式。

12. 日志和排错

程序运行日志写入 MotorEffMAP.log，GUI 底部也会显示实时日志。

现象常见原因处理
提示找不到列配置列名与 Excel 第一行不一致。修改配置，使列名完全一致。
提示 SpeedGrid 必须大于 0网格步长为空、0、负数或非数字。设置为正数，例如 5。
提示网格过大网格步长太小，或包络线异常。增大网格步长，检查数据。
图为空或占比为空有效数据被全部过滤，或效率列无法转换为数字。检查 Excel 数据和配置列名。
编译版找不到配置配置文件不在 exe 同级目录。把 MotorEffMAP.ini 放到 MotorEffMAP.exe 同目录。
中文乱码文件不是 UTF-8 BOM，或被旧编辑器另存为其它编码。用程序配置页保存一次，或用 VSCode 转为 UTF-8 with BOM。
关闭 MAP 后界面仍显示旧图需要确认当前运行入口已读取最新 INI。保存并重载配置，必要时重启程序；当前版本手动视图会重新读取 INI。
构建包体积明显变大PyInstaller 可能使用了全局 Python 或误收集大型库。确认构建日志中的 Python environment 为项目 venv。

13. 维护注意事项

  • 不要把空数据填成 0。空行应保留为 NaN，再按核心列删除。
  • 不要把缺失列静默替换为全 0。缺失列必须明确失败。
  • 面积占比分母按几何运行区域计算，默认从 0rpm / 0Nm 开始。
  • GUI 调用逻辑层时必须捕获可预期的 ValueError，并转成用户提示。
  • 修改 docs/program-implementation.md 后，应同步修改 docs/program-implementation.html。
  • 修改配置字段时，应同步更新 README.md 和本文档配置表。
